# Bug & Incomplete-Feature Audit — 2026-07-02

Scope: full sweep of `crates/bearpaw-api` and `frontend/src`, cross-checked against
`docs/SCANNER_PROTOCOL_REFERENCE.md`, `docs/BC125AT_PROTOCOL.md`, `docs/API_SPEC.md`,
and the real wire captures in `docs/wire_captures/2026-05-21/`.

Baseline at audit time: `cargo test -p bearpaw-api` 63/63 green,
`npm test` 353/353 green, `tsc` clean, eslint 0 errors / 96 warnings.
**A green suite is not a health signal here** — several defects below are pinned
in place by tests that encode the wrong behavior (see C1, M22).

Severity legend:
- **Critical** — corrupts scanner memory / data loss / a core feature does not work at all
- **High** — wedges the app or scanner until restart/power-cycle
- **Medium** — wrong behavior in realistic scenarios, recoverable
- **Low** — polish, dead code, doc drift

## Critical — data corruption and broken core features

### C1. CTCSS code→Hz table is wrong for every code above 64
`crates/bearpaw-api/src/protocol/tones.rs:25-75`

The canonical table (repo's own `docs/BC125AT_PROTOCOL.md` §7.2, decompiled from
Sentinel; also bc125py and bc125csv) is 50 contiguous tones at codes 64–113:
65=69.3, 76=100.0, 78=107.2, 94=171.3, 113=254.1. Bearpaw's table is shifted by
one from code 65 up (65=71.9, 76=103.5), invents "reserved" gaps at 78/79/94/95,
omits seven real tones (69.3, 171.3, 177.3, 183.5, 189.9, 196.6, 199.5 Hz), and
stops at 110 — codes 111–113 decode as "no tone".

- Failure: a channel with CTCSS 100.0 Hz (wire code 76) displays "103.5"; every
  tone from code 65 up is reported wrong.
- The unit tests (`ctcss_reserved_gaps_return_none`, `ctcss_canonical_endpoints`,
  `decode_tone_classifies_kind`) pin the wrong values, and
  `docs/SCANNER_PROTOCOL_REFERENCE.md` §7 repeats the same wrong table with a note
  warning against Sentinel's (correct) mapping. Code, tests, and doc must be fixed
  together.
- Sibling: DCS code 172 (D255) is missing from the DCS map
  (`tones.rs:132-133` jumps 171→252, 173→261); doc has 172→255.

### C2. Channel editing is broken end-to-end: every "Upload Changes" PUT is rejected 422
`frontend/src/api/client.ts:155` / `frontend/src/app/components/views/ChannelsTab.tsx:295-306`
vs `crates/bearpaw-api/src/api/handlers/memory.rs:61-65` and `src/state.rs:117`

`updateChannel` sends `Omit<ChannelData,'index'>`, but the handler deserializes
`Json<ChannelData>` where `index: u16` has **no serde default** — axum rejects with
``missing field `index` `` (422) before the handler runs. Editing any channel and
clicking Upload Changes fails for every channel, every time. Nothing the Channels
tab edits can reach the scanner.

### C3. CIN write path scrambles field order for typical (tone=0/1) channels
`crates/bearpaw-api/src/api/mod.rs:1353-1427` (`write_channel_to_scanner`),
same heuristic in `set_channel_lockout_on_scanner` at `mod.rs:1483-1504`

The wire order (per `parse_cin_response` doc at `protocol/mod.rs:341-345` and the
2026-05-21 capture `CIN,1,Ararat UHF,01451300,AUTO,0,2,0,0`) is
`alpha,freq,mod,tone,delay,lockout,priority` — **no bank field exists on the wire**.
The `has_tone` heuristic (mod.rs:1359-1373) misclassifies every channel whose tone
code is 0 or 1 (tone 0 = "none" is the most common configuration): all four probe
checks pass, `has_tone=false`, and the write emits
`alpha,freq,mod,lockout,delay,priority,bank`.

- Failure: editing a bank-1 channel writes the bank number into the scanner's
  *priority* field, priority into *lockout*, lockout into *tone*. Banks 2–10 put
  an out-of-range value in priority → scanner `ERR` → 400 for the user.
- The lockout toggle picks `lockout_idx=3` (the tone position): "unlock" writes 0
  into the tone field and leaves the real lockout untouched (readback still locked,
  reported as success); "lock" writes invalid tone code 1 → `ERR`.
- Reached from `PUT /memory/channels/{i}`, CSV import (`exports.rs`), and the
  lockout endpoints. `SCANNER_PROTOCOL_REFERENCE.md` §13 item 4 marks this
  heuristic "fixed in Phase 2" — it was fixed only in the read parser, and §4
  explicitly says not to implement CIN writes until write order is verified.

### C4. CIN writes encode the tone as Hz instead of the 0–231 wire code; DCS is silently erased
`crates/bearpaw-api/src/api/mod.rs:1313-1319, 1400`

Reads decode wire code→Hz (`decode_tone`), but writes put `tone_squelch` (Hz) raw
into the wire *code* field via `format_tone_value`. There is no Hz→code encoder
anywhere in the crate.

- Failure: round-tripping an unmodified channel with CTCSS 100.0 Hz writes tone
  *code* 100 — a different tone — silently corrupting squelch on every edit.
- DCS channels: `tone_squelch` is `None` and `tone_dcs_code` is never consulted,
  so any edit writes `0`, erasing the DCS squelch.
- This is pitfall #12 in `SCANNER_PROTOCOL_REFERENCE.md` ("Treating CTCSS field as
  Hz → silently corrupts every channel with a tone").
- Frontend compounds it: the edit sheet validates "Tone must be 0-999"
  (`ChannelEditSheet.tsx:48-51`) and passes the Hz value through unconverted, and
  the frontend `ChannelData` type (`frontend/src/types.ts`) has no `tone_dcs_code`
  field at all, so DCS shows as "—" in the UI.

### C5. `GET /api/v1/lockouts` deadlocks the handler (and every later lockouts call)
`crates/bearpaw-api/src/api/handlers/lockouts.rs:55-58`

Line 55 takes `frequency_lockouts.write()`; line 58 re-locks the same `RwLock`
with `.read()` while the write guard is still alive (shadowing does not drop it —
the old guard lives to end of scope). `std::sync::RwLock` is not reentrant: the
task deadlocks holding the write lock, and every subsequent lockouts request
queues behind it forever. Default `include_frequencies=true`, and the frontend
calls this endpoint.

### C6. Memory sync swallows all transport errors, wholesale-replaces the channel cache, and always reports "Sync complete"
`crates/bearpaw-api/src/api/memory_sync.rs:34-36, 47-49, 66, 94, 101-104`

- Transport errors become empty strings (`unwrap_or_default`); the PRG and EPG
  replies are never validated; `run()` cannot return `Err`, so the "Sync failed"
  branches in `poll.rs:191-201` / `:459-469` are dead code.
- `shadow.channels = channels` replaces the previous cache instead of merging.
- Failure (unplug at channel 100/500): remaining CINs fail, the 400 previously
  good channels vanish, `last_sync` is stamped, UI says "Sync complete".
- Worse (PRG refused, e.g. scanner in a menu): every `CIN,N` returns `NG`, and
  because `parse_cin_response` accepts error replies (see H7), the cache is
  replaced with 500 phantom channels whose alpha tag is "NG".

### C7. Serial multiline read can hang the poll thread forever
`crates/bearpaw-api/src/transport.rs:170-206` (`read_response_multiline`)

Both the `Ok(0)` and `TimedOut` arms `continue` while the buffer is empty; there
is no overall deadline. This runs on the poll thread for **every STS poll**
(`poll.rs:246`). A wedged-but-open port (device stops answering without an I/O
error — e.g. another program left the scanner in PRG, half-dead cable) blocks the
poll thread permanently: every API command times out, no reconnect, no stale flag.
The USB counterpart (`transport_usb.rs:164-188`) correctly errors on
timeout-with-no-data.

## High — wedges the app or the scanner

### H1. Backend restart freezes the UI until the sequence counter catches up
`crates/bearpaw-api/src/api/mod.rs:550` (`sequence: AtomicU64::new(0)`) +
`frontend/src/store/useStore.ts:117-131` (no `lastSequence` reset anywhere)

Backend restarts reseed the WS sequence at 0; the frontend keeps its old
`lastSequence` across reconnects and drops every `state_update` as stale — after
an hour of uptime (~5 Hz), that's ~an hour of frozen "Disconnected" UI on a
healthy backend. Nothing refetches `/status` on reconnect (the initial-load effect
is mount-only) and `handle_socket` sends no snapshot on connect. The documented
example sequence (epoch-millis-shaped, `WEBSOCKET_SCHEMA.md`) suggests the counter
was meant to be clock-seeded. Fix direction: reset `lastSequence` on WS
`connection` transitions (and/or seed the counter from the clock).

### H2. Frontend treats `percent >= 100` as sync completion — races the backend, drops the overlay while PRG is open, and can show zero channels after a full sync
`frontend/src/app/App.tsx:194-241` vs `crates/bearpaw-api/src/api/memory_sync.rs:93-106`

The backend emits `progress(100, "Exiting program mode...")` *before* sending EPG
and *before* writing `shadow.channels`. On that message the frontend flips
`inProgress:false`, immediately fetches `/memory/channels` (gets the pre-sync
shadow — empty on first boot), and then ignores the real "Sync complete" because
`inProgress` is already false. Consequences: after a 30–45 s first sync the app
can still show zero channels / no alpha tags; the auto-sync effect (keyed on
`channels.length === 0`) can kick off a *second* full sync; and the blocking
overlay drops while the PRG bracket is still open — exactly the hazard the
overlay regression guard exists to prevent.

### H3. Sync overlay can lock the app forever if the completion message is lost
`frontend/src/app/App.tsx:192-241` + `crates/bearpaw-api/src/api/handlers/memory.rs:134-141`

`sync.inProgress` is only ever cleared by a WS `progress` message; there is no
sync-status REST endpoint and no re-check on reconnect. If the WS drops in the
last seconds of a sync, "Sync complete" is broadcast into a dead socket and the
full-screen overlay stays up forever. Clicking "Cancel Sync" then hits
`{status:"no_task"}`, which broadcasts nothing — by design only the WS message
flips `inProgress`, so nothing ever clears it short of a reload. (The cancel
handler also toasts "Sync cancelled" unconditionally, even on `no_task`.)

### H4. Three handler paths leak the PRG bracket on early return — scanner stuck in Remote Mode, polling suspended forever
`crates/bearpaw-api/src/api/mod.rs:1103, 1115` (`read_frequency_lockouts_from_scanner`),
`:1326` (`write_channel_to_scanner`), `:1458` (`set_channel_lockout_on_scanner`)

Each `?`-propagates an error after `PRG` succeeded but before `EPG` is sent.
`send_raw_command` sets `program_mode_active=true` on PRG and clears it only on
EPG (`mod.rs:305-358`), so one 3-second CIN timeout leaves (a) the scanner in
"Remote Mode / Keypad Lock" until power-cycle and (b) the poll loop suspended
indefinitely (`poll.rs:241` skips STS/GLG while the flag is set). These paths
bypass `ProgramModeGuard` — the exact leak its doc comment warns about.

### H5. Commands that time out at the HTTP layer still execute later
`crates/bearpaw-api/src/api/mod.rs:313-326`

`send_raw_command` gives up after `recv_timeout(3s)` but the command stays in the
mpsc queue and the poll thread executes it whenever it drains. A timed-out `PRG`
is the worst case: the handler 400s, the inactive guard never sends EPG, and the
queued PRG later puts the scanner into program mode with nothing to take it out —
poll loop hammers STS into `NG` forever. Same mechanism means any timed-out
Hold/Direct executes minutes later after a reconnect.

### H6. `ProgramModeGuard` — drop-order race and `PRG,NG` treated as success
`crates/bearpaw-api/src/api/program_mode.rs:82-96, 100-128`

- Drop clears `program_mode_active` (line 104) *before* EPG is even queued
  (line 117), and EPG only executes on the poll thread's next drain — the poll
  loop can send STS/GLG at a scanner still in PRG mode (the interference the flag
  exists to prevent).
- `enter()` treats any `Ok` transport response as success; `PRG,NG` / `ERR`
  (scanner in menu mode) still yields an "active" guard — subsequent CIN/SCG all
  fail, and drop sends a spurious EPG. CLAUDE.md documents the guard as "waits
  for `PRG,OK`"; it does not classify the reply.

### H7. `parse_cin_response` turns `ERR`/`NG`/desynced replies into phantom channels
`crates/bearpaw-api/src/protocol/mod.rs:350-407`

No reply classification: `CIN,NG` → "NG" becomes the alpha tag →
`Some(ChannelData{alpha_tag:"NG", frequency:0.0, ...})`. An index-mismatched echo
(FIFO desync after a timeout) is also silently mis-parsed — the wrong index
becomes the alpha tag and the alpha tag parses as frequency 0. These fake
channels are cached into shadow state (`handlers/memory.rs:44-48`), served by
`GET /memory/channels`, and exported to CSV. Combined with C6 this can replace
the whole cache with garbage. The parser should return `None` on error replies
and on index mismatch (or callers must run `classify_response` first).

### H8. WS forwarding stalls after a broadcast `Lagged` error
`crates/bearpaw-api/src/api/ws.rs:29-34`

The select arm is `Ok(msg) = rx.recv()`. When `rx.recv()` returns
`Err(RecvError::Lagged)` (channel capacity is 64 at `mod.rs:528`; the poll loop
produces ~5–10 msgs/s), the pattern fails and `tokio::select!` disables that
branch for the rest of the call — the loop then blocks solely on `socket.recv()`.
A quiet browser client (JS WebSockets don't auto-ping) receives no further
updates until it sends a frame or disconnects. Should be `result = rx.recv()`
with the `Err` handled inside the arm.

### H9. GLF lockout walk uses undocumented parameterized commands — at most one global lockout is ever read
`crates/bearpaw-api/src/api/mod.rs:1098-1120`, `handlers/lockouts.rs:106-120`

The documented walk (`BC125AT_PROTOCOL.md:340-348`) is: send bare `GLF`
repeatedly until `GLF,-1`. The code sends `GLF,***` then `GLF,<value>`, which
firmware answers with `ERR`; the walk terminates after the first entry. So
`GET /lockouts` reports one frequency, and `clear_global_lockouts` clears at most
one per call while always reporting `"failed": []` (ULF failures are ignored too).

### H10. Serial transport: no stale-input drain, partial reads accepted, Linux unplug never detected
`crates/bearpaw-api/src/transport.rs:23-37, 87, 127-154`; `transport_usb.rs:151-155`

- `send_inner` never drains the input buffer before writing (only once at open).
  After any mid-response timeout the leftover bytes become the *next* command's
  reply, shifting every subsequent request/response pair — the desync class
  CLAUDE.md pitfalls 2–3 warn about. `UsbTransport::drain_input` exists precisely
  for this; the serial path has no equivalent.
- `Ok(0)` mid-line returns a partial line as success (line 145); the USB path
  similarly returns a truncated buffer on timeout (transport_usb.rs:151-155),
  which then parses with missing fields defaulted instead of being rejected.
- `is_device_gone` matches NotFound/BrokenPipe/etc. but not the `EIO`/`ENXIO`
  a removed `ttyACM`/`ttyUSB` node actually yields on Linux → unplug is classified
  transient, the poll loop warns at 5 Hz forever, and reconnect never fires.

## Medium

### M1. Memory-sync task bookkeeping races
`crates/bearpaw-api/src/api/handlers/memory.rs:103-132`

`sync_task_id` is checked (line 108) and set (line 127) non-atomically around the
queue send (line 119): two concurrent POSTs both start full 30–45 s syncs; a
`ProgramModeGuard::enter` in the send→set window queues a PRG that collides with
the sync bracket; and if the sync fast-fails before the handler's store,
`sync_task_id` is set to a value nothing clears — every future sync returns
`already_running` and every guard 409s until restart.

### M2. No mutual exclusion between the three PRG users
`handlers/memory.rs:143-197` (manual program-mode session) vs
`program_mode.rs:69-71` vs `memory.rs:103-132`

`ProgramModeGuard::enter` checks only `sync_task_id`; `post_memory_sync` checks
neither the guard flag nor a manual session. A settings write or a sync started
during a manual `program-mode/start` session ends with an `EPG` that ejects the
scanner from the user's session while `program_mode_active` stays true — live
display frozen on "Programming", subsequent CIN writes fail.

### M3. `Direct` sets `commanded_mode = Direct` even when the `DO` command fails
`crates/bearpaw-api/src/api/poll.rs:175-186, 443-453`

Hold/Scan gate on `response.is_ok()`; Direct doesn't, and the `DO` reply is never
classified (`ERR`/`NG` counts as success). Currently unreachable because nothing
routes to it (see Incomplete Features), but latent.

### M4. Settings setters truncate before range-checking
`handlers/settings.rs:105` (`set_battery`: `as u64 → as u8` then `1..=16`), same
pattern at `:162` (`set_key_beep`), `:221` (`set_priority`), `:274`
(`set_search`), `:766` (`set_contrast`)

`{"charge_time": 257}` truncates to 1, passes validation, and writes `BSV,1` —
200 OK for a value the client never sent.

### M5. Comma injection through `modulation` into the CIN wire command
`handlers/memory.rs:84` + `mod.rs:1391-1427`; same hole in `import_csv`

`validate_wire_field` (security.rs:90-98) only blocks control characters.
`write_channel_to_scanner` comma-sanitizes `alpha_tag` but splices `modulation`
in raw: `"modulation": "FM,0,0,1,0"` injects extra wire fields, shifting
delay/lockout/priority.

### M6. Settings GET handlers cache scanner errors as real values
`handlers/commands.rs:310-320` (`get_squelch`), `settings.rs:78-82, 193-197,
742-746`, `mod.rs:1140-1247` (`read_settings_snapshot_from_scanner`)

An `NG`/`ERR`/garbage reply parses to 0/defaults, is returned as 200, and is
persisted into the settings snapshot — one flaky read permanently overwrites
cached settings with zeros.

### M7. `post_lockout` accepts any channel index and can echo error text back onto the wire
`handlers/commands.rs:142-240`, `mod.rs:1448-1519`

No `1..=500` check. Channel 0 → `CIN,0` → `ERR` → the code pads the parts array
and writes `CIN,0,ERR,,,,1,` back to the scanner; the temporary-lockout map is
polluted with the bogus index on the fallback path. (`GET /memory/channels/{i}`
similarly has no range check — `CIN,501` surfaces as a phantom channel per H7.)

### M8. STS tail signature rejects documented firmware variants
`crates/bearpaw-api/src/protocol/mod.rs:194-224`

The anchor requires numeric SIG_LVL at tail offset 6 **and** numeric BK_DIMMER at
offset 8. The reference doc's own firmware-1.04.02 example ends `...,1,,` (empty
BK_DIMMER) and fails the signature, as does the 6-field tail documented in
`BC125AT_PROTOCOL.md:856-874` — on such firmware every STS poll drops and
squelch relies solely on GLG. Only the 1.06.06 capture shape parses.

### M9. WS-subscribe effect cleanup cancels in-flight resume timers on tab switch / connection blip
`frontend/src/app/App.tsx:112-115, 249-254` (deps at 266-275)

The effect re-runs when `currentTab` or `connected` changes; cleanup clears
`scanResumeTimerRef` and can strand `scanResumeInFlightRef=true` if it fires in
the 250 ms release window — every later scan-resume silently no-ops for the
session. A pending 1 s delayed resume (lockout flow) is likewise cancelled by a
tab switch, leaving the scanner stuck in HOLD.

### M10. Mount-time `getStatus()` bypasses the sequence check
`frontend/src/app/App.tsx:296-299`

The REST snapshot is applied with no sequence; if WS updates arrived first, the
older snapshot overwrites newer state until the next poll tick.

### M11. `ScannerWebSocket.disconnect()` cannot cancel a pending reconnect timer
`frontend/src/websocket/ScannerWebSocket.ts:26, 66-71, 86-104`

The timer id isn't stored; a surviving timer calls `connect()`, which
unconditionally re-arms `shouldReconnect` — a zombie socket reconnects after an
explicit disconnect. Latent today (provider lives for app lifetime, no
StrictMode), bites any unmount/remount and tests.

### M12. Keyboard shortcuts: fire while typing, hijack copy, no error handling, dead key codes
`frontend/src/hooks/useKeyboardShortcuts.ts:19-80`

- No editable-element check: Ctrl/Cmd+C `preventDefault()`s and overwrites the
  clipboard with the live frequency even with text selected in an input; Ctrl+S/H
  fire scanner commands mid-typing.
- `api.sendScan()/sendHold()/sendKey()/toggleTemporaryLockout()` have no `.catch`
  → unhandled rejections, zero user feedback.
- Ctrl+↑/↓ call `sendKey('UP'/'DOWN')`, but the backend allowlist
  (`commands.rs:85-88`) accepts only single-char codes → 400 on every press,
  despite the shortcuts toast advertising them.
- The advertised "?" shortcut is unreachable (modifier check precedes the switch).

### M13. Channel edit sheet: Cancel keeps edits; delay/frequency/tone validation wrong
`frontend/src/app/components/views/ChannelEditSheet.tsx:41-51, 243-248`,
`ChannelsTab.tsx:339-348`

- Every keystroke writes into the store draft; Cancel only closes the sheet — the
  "cancelled" edit uploads with the next Upload Changes.
- Delay validation allows 0–30; backend accepts only {-10, -5, 0..5} — legal
  pre-delays can't be saved, illegal delays 400 at upload.
- Frequency validation treats 25–512 MHz as contiguous; the radio covers
  25–54 / 108–174 / 225–380 / 400–512 — gap frequencies fail only at the wire.
- Tone field edits pass Hz straight through (see C4).

### M14. Channels tab: bank-order cache freezes an empty order; drag-reorder never resets after upload
`frontend/src/app/components/views/ChannelsTab.tsx:211-227, 390-515`

- If the tab mounts before channels load (deep link during initial sync),
  `bankOrders[bank]` is seeded `[]` and — `[]` being non-nullish — stays empty
  forever: "No channels match your filters" even after sync; channels added later
  (CSV import) are silently dropped from the list.
- After an upload, `bankOrders`/`reorderTargets` keep old→new mappings and moved
  channels stay permanently "pending"; a second Upload Changes re-writes each
  channel's current content to its shifted slot — scrambling channel memory.

### M15. Device tab defects
`frontend/src/app/components/views/DeviceTab.tsx`

- Service Search: state initialized with 6 booleans, UI renders 8 toggles, the
  backend requires exactly 10 (`settings.rs:461-463`) → toggling before settings
  load (or after a failed load) 400s every time; groups 9–10 are never exposed.
- Contrast slider allows 0; backend requires 1–15 → dragging to 0 always errors.
- "Select Page" uses `lockedChannels` (all) instead of the filtered list —
  "Unlock Selected" unlocks channels hidden by the active filter.
- MQTT preferences saved under camelCase keys but loaded as snake_case
  (`DeviceTab.tsx:60-76` vs `App.tsx:151-156`) — MQTT settings silently reset on
  every restart.

### M16. Activity export: silently truncated and off-by-a-day
`frontend/src/app/components/views/ActivityExportSheet.tsx:80-83, 99`

- No `limit` passed; `analytics_activity_log` defaults to 100 — an "All Time"
  export of months of history contains the 100 most recent hits, no warning.
- Custom range: `end_time` is midnight at the *start* of the end day (parsed as
  UTC), excluding that day's hits; an unset date sends `start_time=0&end_time=0`
  → guaranteed-empty CSV. Failures only `console.error` — no toast.

### M17. `useActivityLogTracker` consumes `state_update` without the sequence check
`frontend/src/hooks/useActivityLogTracker.ts:29-63`

Out-of-order updates the store correctly drops still drive the hit state machine —
reordered squelch transitions can log spurious or truncated hit entries.

### M18. Sequence take/send not atomic across producers
`crates/bearpaw-api/src/api/poll.rs:792` + `handlers/commands.rs:292`

The poll thread and the volume handler each `fetch_add` then send unsynchronized;
messages can hit the wire in inverted order and the frontend drops the
lower-sequence (full) update for that tick.

### M19. `FitText` crashes when `ResizeObserver` is missing — and its test asserts the crash
`frontend/src/app/components/FitText.tsx:62`,
`frontend/src/app/components/__tests__/FitText.test.tsx:45-51`

`new ResizeObserver(fit)` is unguarded; a missing constructor unmounts the whole
tree. The test is titled "does not crash when ResizeObserver is missing" but
asserts `expect(() => render(...)).toThrow()` — documenting the defect.

### M20. Poll-thread panic leaves the app reporting "connected" forever
`crates/bearpaw-api/src/api/poll.rs:47-55`

Only an `Err` *return* flips `connection_status`; a panic (e.g. via the
`.unwrap()`s on the `sync_task_id` mutex, poisonable from handler panics) kills
the thread silently — UI shows healthy, every command times out.

### M21. Misc backend edges
- `UsbTransport::open` (`transport_usb.rs:62`): `device_descriptor()?` inside the
  enumeration loop — one unrelated flaky USB device fails the whole open
  (`probe_known_scanner_via_usb` in config.rs does it right with `continue`).
- `load_config` (`config.rs:105-118`): YAML parse errors silently become
  `Config::default()` — a typo discards explicit `device.port`/`usb_vid` with no
  log. `usb_vid` without `usb_pid` (or vice versa) is silently ignored
  (config.rs:143).
- `put_memory_channel` never validates `frequency` (memory.rs:61-95;
  `validate_frequency` exists unused); `import_csv` accepts 25–1300 MHz vs the
  device's 512 MHz cap.
- `ULF` frequency not zero-padded to 8 digits (`lockouts.rs:114`).
- `day_hour` (`mod.rs:1057-1064`) buckets the hourly heatmap in UTC while
  claiming local-time parity — analytics shifted by the UTC offset.
- `importCsv` error path (`frontend/src/api/client.ts:396`) does
  `await response.json()` with no text fallback — an HTML 502 throws
  `SyntaxError` instead of an `APIError`.
- `useShellStatusText` (`hooks/useShellStatusText.ts:50-58`): if the component
  unmounts before `subscribeBackendStatus` resolves, the listener is never
  unsubscribed.

## Incomplete features & dead code

1. **Direct tune (`POST /frequency`) is documented but unrouted.** The full
   plumbing exists as dead code — `FrequencyRequest`, `validate_frequency`/
   `validate_modulation` (control.rs:34-69), `ControlCommand::Direct`, the poll
   `DO` handler — but no route constructs it. (API_SPEC.md:314)
2. **`GET /health` documented (API_SPEC.md:161, openapi.json) but not
   implemented**; a frontend contract test references it.
3. **MQTT panel configures a feature that doesn't exist** — the backend has only
   preference defaults; no MQTT publisher anywhere. (`DeviceTab.tsx:1529-1626`)
4. **Dead Preferences controls** (`DeviceTab.tsx`): "Auto-Connect on Startup" and
   "Check for Updates" switches are `defaultChecked` with no handler (backend
   prefs exist but are unwired); "Audio Output Device" lists hardcoded fake
   devices; "Recording Buffer Size" does nothing; "Data Retention" shows "Keep
   Forever" while the backend's daily cleanup deletes hits older than 30 days
   regardless (`api/mod.rs:482-498`).
5. **Hardcoded version "v2.4.0-beta"** (`DeviceTab.tsx:1347`; real version 1.1.1)
   and placeholder `bearpaw-scanner.*` links using raw `window.open` instead of
   the Tauri-safe helper.
6. **`banks_update` WS broadcast has no frontend subscriber** — backend broadcasts
   on every bank change "so the UI can mirror reality" (`ws.rs:47-55`); nothing
   listens. Second-client/backend-initiated bank changes never reach the UI.
7. **Dead WS ping handler / no heartbeat** (`ScannerWebSocket.ts:41-44`): the
   client answers server pings the backend never sends; a silently dropped TCP
   connection shows "connected" with a frozen display until OS timeout.
8. **`DeviceConfig.transport` parsed but never read** (`config.rs:48`).
9. **`validate_channel_name` has no caller** (`protocol/mod.rs:445`) despite CIN
   writes existing; **`GlgFrame.tone_code` is parsed but never surfaced** — live
   tone display is unimplemented.
10. **Channel memory is not persisted** — CLAUDE.md says "Persistence to SQLite in
    state.rs", but SQLite holds only preferences/analytics; every restart needs a
    full 30–45 s re-sync. Doc/feature mismatch.
11. **No time-based staleness**: `live.stale` flips only on hard disconnect
    (`poll.rs:610-611`); persistent parse failures freeze the display silently
    with `stale:false` and no `state_stale` event.

## API spec drift (docs/API_SPEC.md vs router)

- `DELETE /memory/sync/{task_id}` documented; implementation is
  `POST /memory/sync/cancel` with a different response shape.
- `GET /memory/channels` documented as `{channels, count, last_sync}`; handler
  returns a bare array (frontend expects the array — the spec is stale).
- `POST /memory/sync` documented 202/409; returns 200 with `"already_running"`.
- `POST /commands/key` spec codes (`UP`, `DOWN`, `FUNC`, `MENU`, `L_OUT`) don't
  exist in the allowlist (single-char codes only).
- `GET /status` documented to 503 when disconnected; always 200.
- `POST /api/v1/preferences` routes to `reset_preferences` — an undocumented,
  destructive alias for a natural client mistake.
- Implemented but undocumented: `/banks`, `/volume`, `/squelch`, `/config`,
  `/settings/*` (11), `/lockouts` (+3 clear), `/memory/program-mode/start|end`,
  `/analytics/*`.
